# CLAUDE.md — เว็บแอปสมุดวางแผนการดูแลชีวิตระยะท้าย (Advance Care Planning Web App)

> ชื่อโปรเจกต์ชั่วคราว: `acp-booklet` (ชื่อแบรนด์จริงจะกำหนดภายหลัง — เขียนโค้ดให้เปลี่ยนชื่อได้ง่าย เก็บชื่อไว้ใน config เดียว)

## ภาพรวมโปรเจกต์

เว็บแอปให้คนทั่วไปเขียน "หนังสือแสดงเจตนาเกี่ยวกับการดูแลสุขภาพช่วงท้ายของชีวิต" (Advance Care Planning / Living Will ตามแนวมาตรา 12 พ.ร.บ.สุขภาพแห่งชาติ พ.ศ. 2550) ผ่านฟอร์มแบบ step-by-step แล้วสร้างเป็นไฟล์ PDF ภาษาไทยที่สวยงาม พร้อมปริ้นไปเซ็นชื่อกับพยาน

**หลักการสูงสุด: ข้อมูลไม่ออกจากเครื่องผู้ใช้เด็ดขาด**
- ไม่มี backend, ไม่มี database, ไม่มี analytics ที่เก็บเนื้อหาฟอร์ม
- ประมวลผลและสร้าง PDF ในเบราว์เซอร์ 100%
- Deploy เป็น static site (Cloudflare Pages / GitHub Pages / Netlify)

## Tech Stack

- React 18 + Vite + TypeScript (strict mode)
- Tailwind CSS
- สร้าง PDF: `pdf-lib` + `@pdf-lib/fontkit` (ฝังฟอนต์ไทย) — ห้ามใช้วิธี render HTML เป็นภาพ (html2canvas) เพราะไฟล์ใหญ่ ข้อความ search/copy ไม่ได้
- ฟอนต์ไทย: Sarabun (Regular, Bold) จาก Google Fonts — ดาวน์โหลดไฟล์ .ttf มาไว้ใน `/public/fonts/` และฝังลง PDF ตอน generate
- State: React state + Context ก็เพียงพอ (แอปเดียว ฟอร์มเดียว) — ไม่ต้องใช้ Redux/Zustand เว้นแต่จำเป็นจริง
- ไม่ใช้ localStorage/sessionStorage เก็บเนื้อหาฟอร์มโดย default (ข้อมูลอ่อนไหว + เครื่องสาธารณะ) — ใช้ in-memory state และให้ผู้ใช้ export เป็นไฟล์เอง

## โครงสร้างเนื้อหา

เนื้อหาฟอร์มทั้งหมดอยู่ในไฟล์ `acp-booklet-structure.md` (อยู่ใน repo เดียวกัน) — อ่านไฟล์นั้นก่อนเริ่มงาน ประกอบด้วย 8 ส่วน:

1. ข้อมูลของฉัน
2. สิ่งที่ฉันให้คุณค่า
3. การรักษาในวาระสุดท้ายของชีวิต (หัวใจตามมาตรา 12)
4. การดูแลแบบประคับประคอง
5. ผู้ตัดสินใจแทนฉัน
6. หลังจากฉันจากไป
7. ข้อความจากใจ
8. การลงนาม (ช่องเซ็นชื่อ + พยาน 2 คน — อยู่ใน PDF เท่านั้น ไม่ต้องมีในฟอร์มเว็บ)

**แยกเนื้อหาออกจากโค้ด**: นิยามคำถามทั้งหมดเป็น data (TypeScript object/JSON schema) ใน `src/content/questions.ts` — ห้าม hardcode ข้อความคำถามในคอมโพเนนต์ เพื่อให้แก้เนื้อหา/แปลภาษาได้โดยไม่แตะ UI

## User Flow

```
หน้าแรก (อธิบายว่าคืออะไร + คำมั่นเรื่องความเป็นส่วนตัว)
  → เริ่มทำ / เปิดแบบร่างจากไฟล์ (.json)
  → Wizard 7 ขั้น (ส่วนที่ 1–7) มี progress bar
  → หน้าสรุป/ตรวจทาน (แสดงทุกคำตอบ แก้ไขย้อนได้)
  → ดาวน์โหลด PDF + ดาวน์โหลดแบบร่าง (.json)
  → หน้า "ขั้นตอนถัดไป" (เซ็นชื่อ, พยาน, แจ้งครอบครัว, มอบสำเนาแพทย์)
```

กติกาสำคัญของฟอร์ม:
- **ทุกข้อข้ามได้** ไม่มี required field ยกเว้นชื่อ-นามสกุล (ขั้นต่ำสุดที่เอกสารต้องมี — ยกเว้นกรณีติ๊ก "ปริ้นไปเขียนเอง")
- ข้อที่ข้าม เมื่ออยู่ใน PDF ให้เว้นเป็นเส้นประ/ช่องติ๊กว่างสำหรับเขียนด้วยปากกา (ตัดสินใจ 2026-07-08 — เดิมพิมพ์ "— ยังไม่ระบุ —" แต่เปลี่ยนเพื่อรองรับผู้ที่อยากเขียนบางข้อด้วยมือ)
- ส่วนที่ 1 มี checkbox "ต้องการปริ้นไปเขียนข้อมูลส่วนตัวเอง" (webOnly ไม่พิมพ์ลง PDF) — ติ๊กแล้วช่องข้อมูลส่วนตัวถูกซ่อน และใน PDF เว้นเส้นให้เขียนมือทั้งหมด สำหรับคนกังวลเรื่องความเป็นส่วนตัว
- หน้าแรกมีปุ่ม "ดาวน์โหลดแบบฟอร์มเปล่า (PDF)" — ทุกช่องเว้นให้เขียนด้วยปากกาทั้งฉบับ
- ปุ่ม "บันทึกแบบร่าง" มีทุกหน้า → ดาวน์โหลดไฟล์ `.json` (schema versioned: `{ "version": 1, "data": {...} }`)
- ออกจากหน้า/refresh ระหว่างทำ → มี beforeunload warning
- ไม่มีระบบ account/login ใด ๆ

## PDF Output

- ขนาด A4, ระยะขอบมาตรฐานพร้อมปริ้น, ฟอนต์ Sarabun ฝังในไฟล์
- หน้าปก: ชื่อเอกสาร + ชื่อเจ้าของ + วันที่สร้าง
- เนื้อหาตามลำดับ 8 ส่วน จัดหน้าอ่านง่าย มีหัวข้อชัด
- ส่วนที่ 3 ต้องมีข้อความเงื่อนไขนำ (มีผลเฉพาะเมื่อแพทย์วินิจฉัยว่าอยู่ในวาระสุดท้าย) พิมพ์เด่นชัด
- หน้าสุดท้าย: ช่องลงนามเจ้าของ + พยาน 2 คน + ผู้ตัดสินใจแทน (เส้นให้เซ็น, ชื่อตัวบรรจง, วันที่)
- ภาคผนวก: คำแนะนำหลังปริ้น + คำอธิบายมาตรา 12 ย่อ + ช่อง "ทบทวนล่าสุดเมื่อ" + disclaimer
- Footer ทุกหน้า: เลขหน้า X/Y + ชื่อเจ้าของเอกสาร (กันหน้ากระดาษปะปนกัน)
- ตั้งชื่อไฟล์: `acp-{ชื่อ}-{YYYYMMDD}.pdf`

## ทิศทางดีไซน์

โจทย์อารมณ์: **สงบ อบอุ่น มีศักดิ์ศรี** — นี่คือเอกสารที่คนเขียนถึงคนที่เขารัก ไม่ใช่ฟอร์มราชการ และไม่ใช่เว็บงานศพ

- โทนสี: หลีกเลี่ยงดำ-ขาว-เทาแบบไว้อาลัย และหลีกเลี่ยงชุดสีสำเร็จรูปยอดนิยม (cream + terracotta) — เสนอแนวทาง: โทนธรรมชาติเวลาเช้า เช่น เขียวใบชาอ่อน / ฟ้าหม่นก่อนรุ่ง / ครีมกระดาษสา ตัดด้วยสีเข้มหนึ่งสีสำหรับข้อความ เลือกแล้วนิยามเป็น design token ใน Tailwind config
- Typography: ฟอนต์ไทยที่อ่านสบายบนจอ (เช่น Sarabun หรือ IBM Plex Sans Thai สำหรับ UI) — ให้ตัวหนังสือเป็นพระเอก ไม่ต้องมี illustration เยอะ
- Signature element หนึ่งอย่างพอ: เช่น progress indicator ที่ออกแบบเป็นเส้นทาง/แม่น้ำที่ค่อย ๆ เดินไปถึงปลายทาง — เลือกหนึ่งอย่างแล้วทำให้ประณีต ที่เหลือให้เรียบและนิ่ง
- Microcopy สำคัญมาก: น้ำเสียงนุ่ม ชวนคิด ไม่สั่ง เช่น ปุ่มข้ามใช้คำว่า "ขอคิดดูก่อน" แทน "ข้าม", ทุก step มีประโยคเกริ่นสั้น ๆ ว่าทำไมคำถามนี้มีความหมาย
- Responsive ถึงมือถือ (คนกลุ่มเป้าหมายจำนวนมากใช้มือถือเป็นหลัก), รองรับ keyboard navigation, focus state ชัดเจน, respect `prefers-reduced-motion`
- ขนาดตัวอักษร UI เริ่มต้นใหญ่กว่าปกติเล็กน้อย (ผู้ใช้สูงวัยเยอะ) และมีปุ่มปรับขนาดตัวอักษร

## โครงสร้างโปรเจกต์ (แนวทาง)

```
src/
  content/questions.ts      # นิยามคำถามทั้งหมด (single source of truth)
  content/pdfText.ts        # ข้อความคงที่ใน PDF (เงื่อนไขนำ, disclaimer, ภาคผนวก)
  types/form.ts             # FormData types + JSON draft schema (versioned)
  components/wizard/        # Step components, ProgressBar, Navigation
  components/fields/        # ChoiceField, TextField, MultiSelectField, PersonField
  components/review/        # หน้าตรวจทาน
  lib/pdf/generator.ts      # pdf-lib: layout engine ง่าย ๆ + font embedding
  lib/draft.ts              # export/import JSON draft + validation + version migration
  pages/                    # Home, Wizard, Review, Done
public/fonts/               # Sarabun-Regular.ttf, Sarabun-Bold.ttf
```

## สิ่งที่ห้ามทำ

- ห้ามส่งข้อมูลฟอร์มไปที่ endpoint ใด ๆ (รวมถึง error tracking ที่แนบ payload)
- ห้ามใช้ CDN โหลดฟอนต์ตอน runtime สำหรับ PDF — ฟอนต์ต้อง bundle มากับแอป
- ห้ามเก็บเนื้อหาฟอร์มใน localStorage/sessionStorage/cookies
- ห้ามใส่คำแนะนำทางการแพทย์หรือกฎหมายเกินกว่าที่ระบุใน content files
- ห้ามใช้ภาพ/ไอคอนที่สื่อความตายตรง ๆ (โลงศพ หลุมศพ ริบบิ้นดำ)

## คุณภาพขั้นต่ำก่อนถือว่าเสร็จ

- [ ] ทำครบ flow: กรอก → ตรวจทาน → ได้ PDF ภาษาไทยที่ปริ้นออกมาสวยจริงบน A4
- [ ] สระ/วรรณยุกต์ไทยใน PDF แสดงถูกต้อง ไม่ลอย ไม่ซ้อน (ทดสอบคำเช่น "น้ำใจ", "ที่สุด", "ปู่ย่า")
- [ ] Export แบบร่าง → refresh → import กลับมา ข้อมูลครบทุก field
- [ ] ข้ามทุกข้อแล้ว generate PDF ได้ ไม่ error และทุกข้อเว้นเป็นเส้นประ/ช่องติ๊กว่างให้เขียนด้วยปากกา
- [ ] ข้อความยาวมาก (เช่น ข้อความจากใจ 2,000 ตัวอักษร) ตัดหน้า PDF ถูกต้อง
- [ ] ใช้งานบนมือถือได้จริงทั้ง flow รวมถึงดาวน์โหลดไฟล์
- [ ] Lighthouse accessibility ≥ 95
- [ ] `npm run build` ผ่าน, ไม่มี TypeScript error, bundle ไม่บวมเกินจำเป็น

## ลำดับการพัฒนา (Milestones)

1. **M1 — โครง + เนื้อหา**: ตั้งโปรเจกต์, สร้าง `questions.ts` จาก `acp-booklet-structure.md`, wizard เดินหน้า-ถอยหลังได้, state ครบ
2. **M2 — PDF engine**: generate PDF ภาษาไทยจาก state ได้ครบทุกส่วน รวมหน้าลงนามและภาคผนวก
3. **M3 — Draft save/load**: export/import JSON + validation + beforeunload warning
4. **M4 — Polish**: ดีไซน์ตามทิศทางข้างบน, microcopy, responsive, accessibility, หน้า "ขั้นตอนถัดไป"
5. **M5 — Deploy**: static build + deploy + ทดสอบปริ้นจริงจากหลาย browser (Chrome, Safari mobile)

เริ่มจาก M1 และหยุดถามเมื่อเจอการตัดสินใจที่กระทบเนื้อหาหรือกฎหมาย (เช่น ถ้อยคำในส่วนที่ 3 และหน้าลงนาม) — เรื่องเทคนิคตัดสินใจเองได้เลย
